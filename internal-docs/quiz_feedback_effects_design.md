# クイズ演出ライブラリ方針メモ

更新日: 2026-03-21

## 現在の採用方針

- 現在の回答演出は `flutter_animate` を前提にしている
- 理由:
  - 既存 widget に対して差分実装しやすい
  - `fade` `scale` `slide` `shake` などの基本演出がそろっている
  - `onComplete` ベースで演出終了後の処理を組みやすい
  - 専用アセット制作なしで運用できる

## 切り替え判断の基準

### `flutter_animate` を継続するケース

- 演出の中心が widget の見た目変化で完結する
- 正解/不正解ごとの分岐をコードで細かく制御したい
- 画面上の選択肢や overlay を既存レイアウトのまま強化したい

### `lottie` へ切り替えまたは併用するケース

- デザイナー制作の完成アニメーションをそのまま再生したい
- 成功時や失敗時にブランド感の強い演出を追加したい
- 画面全体のカットイン演出をアセット主導で管理したい

注意:

- `lottie` 単体では選択肢ハイライトや画面内の局所演出は置き換えにくい
- 実際には `flutter_animate` と併用する構成になりやすい

### `rive` へ切り替えまたは併用するケース

- インタラクティブな高品質アニメーションが必要
- 状態遷移をアセット側の state machine で持ちたい
- 演出表現の優先度が高く、専用制作フローを維持できる

注意:

- 導入コストと制作コストが最も高い
- 現在のクイズ演出規模ではオーバースペックになりやすい

## 拡張方針

### 軽量な強化

- 正解演出をもう少し派手にしたい場合は `confetti` の追加を検討する
- この場合も基本制御は `flutter_animate` のまま維持する

### 中規模の強化

- 中央バッジや画面全体フラッシュを `lottie` 化する
- 選択肢のハイライトやレイアウト連動は `flutter_animate` に残す

### 大規模な強化

- 演出アセットをデザイン主導に寄せるなら `rive` を再評価する
- その場合は quiz 画面固有の演出層を分離して、widget ロジックとアセット制御の責務を切る

## 保守メモ

- 回答後の進行制御はライブラリ依存ではなくアプリ側の状態管理で持つ
- そのため、演出ライブラリを差し替える場合も「演出完了後に処理を進める」という契約は維持する
- 将来差し替える場合は、まず overlay 部分と選択肢演出部分を別 widget に寄せてから進めるのが安全
